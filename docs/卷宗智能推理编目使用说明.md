# file_directory_predictor.py 中文使用说明

[English usage guide](Smart_Case_Filing_Usage.en.md)

本文档说明如何安装、配置和运行 `file_directory_predictor.py`，以及如何理解输出结果和排查常见问题。

`file_directory_predictor.py` 是一个法院卷宗材料目录推测工具。它读取输入文件内容，调用文本模型和视觉模型生成文档画像，再结合 `catalog-mapping.xlsx` 编目规则表，推测文件应归属的目录。

最终目录格式：

```text
案件类型 / 卷宗 / 二级目录 / 材料类别
```

## 1. 环境要求

### 1.1 Python

建议使用 Python 3.10 或更高版本。

检查版本：

```bash
python --version
```

### 1.2 Python 依赖

安装基础依赖：

```bash
pip install pandas openpyxl pymupdf python-docx requests
```

依赖用途：

| 依赖 | 用途 |
| --- | --- |
| `pandas` | 读取 Excel 编目规则和表格文件 |
| `openpyxl` | 支持 `.xlsx` 文件读取 |
| `pymupdf` | PDF 文本提取和页面渲染 |
| `python-docx` | `.docx` 文档内容提取 |
| `requests` | 调用 OpenAI-compatible HTTP 模型服务 |

### 1.3 LibreOffice（可选）

只有处理 `.doc` 老 Word 文件时才需要。

```bash
libreoffice --version
```

如果不处理 `.doc` 文件，可以不安装 LibreOffice。

## 2. 模型服务配置

脚本优先使用 OpenAI-compatible HTTP API。最少需要配置：

- `base_url`
- `api_key`
- `model`

### 2.1 通用配置

Bash:

```bash
export AI_BASE_URL="https://api.openai.com/v1"
export AI_API_KEY="sk-..."
export AI_MODEL="gpt-4.1"
```

PowerShell:

```powershell
$env:AI_BASE_URL = "https://api.openai.com/v1"
$env:AI_API_KEY = "sk-..."
$env:AI_MODEL = "gpt-4.1"
```

### 2.2 分别配置文本模型和视觉模型

如果文本和视觉使用不同服务或不同模型，可以使用专用变量。

Bash:

```bash
export AI_CHAT_BASE_URL="https://api.deepseek.com/v1"
export AI_CHAT_API_KEY="sk-..."
export AI_CHAT_MODEL="deepseek-chat"

export AI_VISION_BASE_URL="https://api.openai.com/v1"
export AI_VISION_API_KEY="sk-..."
export AI_VISION_MODEL="gpt-4.1"
```

PowerShell:

```powershell
$env:AI_CHAT_BASE_URL = "https://api.deepseek.com/v1"
$env:AI_CHAT_API_KEY = "sk-..."
$env:AI_CHAT_MODEL = "deepseek-chat"

$env:AI_VISION_BASE_URL = "https://api.openai.com/v1"
$env:AI_VISION_API_KEY = "sk-..."
$env:AI_VISION_MODEL = "gpt-4.1"
```

### 2.3 环境变量优先级

```text
chat:
  AI_CHAT_BASE_URL / AI_CHAT_API_KEY / AI_CHAT_MODEL
  -> AI_BASE_URL / AI_API_KEY / AI_MODEL

vision:
  AI_VISION_BASE_URL / AI_VISION_API_KEY / AI_VISION_MODEL
  -> AI_BASE_URL / AI_API_KEY / AI_MODEL
```

### 2.4 z-ai fallback

如果没有配置完整的 HTTP 三件套，脚本会回退到旧的 `z-ai` CLI：

```bash
z-ai chat
z-ai vision
```

使用 fallback 前，请确认：

```bash
z-ai chat -p "测试"
z-ai vision -p "描述图片内容" -i /path/to/image.png
```

## 3. 编目规则表

默认使用 `catalog-mapping.xlsx`。该文件至少需要包含以下列：

```text
case_type
volume
second_level_directory
constraint
material_category
catalog_name_example
```

运行时建议显式传入：

```bash
--catalog ./catalog-mapping.xlsx
```

## 4. 支持的输入文件

| 文件类型 | 扩展名 | 处理方式 |
| --- | --- | --- |
| PDF | `.pdf` | 提取文本，并把前 N 页渲染为图片交给视觉模型 |
| Word 新格式 | `.docx` | 提取段落和表格文本 |
| Word 老格式 | `.doc` | 先用 LibreOffice 转为 `.docx` |
| 图片 | `.png` `.jpg` `.jpeg` `.gif` `.webp` `.bmp` | 交给视觉模型做识别和 OCR |
| 表格 | `.xlsx` `.xlsm` `.xls` | 读取所有 sheet 的前 200 行 |
| 文本 | `.txt` `.csv` `.tsv` `.md` `.json` `.log` `.xml` `.html` `.htm` | 尝试多编码读取 |
| 其他 | 任意扩展名 | 兜底按文本读取 |

## 5. 单文件预测

基本命令：

```bash
python file_directory_predictor.py ./sample.pdf --catalog ./catalog-mapping.xlsx
```

JSON 输出：

```bash
python file_directory_predictor.py ./sample.pdf --catalog ./catalog-mapping.xlsx --json
```

保存结果和日志到指定文件：

```bash
python file_directory_predictor.py ./sample.pdf --catalog ./catalog-mapping.xlsx --json --output ./result.json --log ./run.log
```

如果不显式指定 `--output` 和 `--log`，程序会默认保存到脚本所在目录：

- `file_directory_predictor_output.txt`：运行结果，也就是原本输出到 stdout 的内容
- `file_directory_predictor.log`：运行日志，也就是原本输出到 stderr 的内容

输出内容包括：

- 文件路径
- 文件类型
- 推测目录
- 编目名称示例
- 置信度
- 匹配理由
- VLM 视觉分析结果
- LLM 文本分析结果
- 命中的编目规则

## 6. 批量预测

处理目录下所有一级文件：

```bash
python file_directory_predictor.py --batch ./input-files --catalog ./catalog-mapping.xlsx
```

批量 JSON 输出：

```bash
python file_directory_predictor.py --batch ./input-files --catalog ./catalog-mapping.xlsx --json
```

注意：当前批量模式只处理指定目录下的直接文件，不递归处理子目录。

## 7. 批量可恢复智能体模式

智能体模式适合需要保留执行轨迹、低置信复核、批量运行摘要和断点恢复的编目任务。该模式使用 OpenAI-compatible HTTP API 配置和旧 `z-ai` fallback，与普通预测共用同一套提取、视觉分析、文本分析和候选目录规则。

```bash
python file_directory_predictor.py ./sample.pdf \
  --catalog ./catalog-mapping.xlsx \
  --agent \
  --trace ./logs/sample.trace.jsonl \
  --review-output ./logs/sample.review.json \
  --json
```

智能体模式仍会保持普通 JSON 输出字段兼容，并额外输出 `agent_state`、`trace`、`review_output`、`resume` 等字段。

批量智能体示例：

```bash
python file_directory_predictor.py \
  --batch ./input-files \
  --catalog ./catalog-mapping.xlsx \
  --agent \
  --trace ./agent-runs/demo \
  --review-output ./agent-runs/demo/reviews \
  --json
```

批量模式只处理指定目录下的直接文件，不递归子目录。`--trace` 在批量智能体模式下会作为运行目录使用；`--review-output` 如果指向目录，会作为复核包和 `index.json` 的输出目录。运行完成后生成：

```text
agent-runs/<run_id>/
  manifest.json
  traces/<file_id>.trace.jsonl
  reviews/<file_id>.review.json
  reviews/index.json
  outputs/<file_id>.json
```

参数说明：

| 参数 | 含义 |
| --- | --- |
| `--agent` | 启用智能体状态机。 |
| `--trace <path>` | 单文件模式下写入 JSONL trace；批量模式下作为 run directory。 |
| `--review-output <path>` | 单文件模式下是复核 JSON 文件；批量模式下是 review 目录。 |
| `--resume <trace-or-manifest>` | 从单文件 trace 或批量 manifest 恢复；终态不重跑，非终态尝试从下一步继续。 |
| `--agent-retry-attempts <n>` | 智能体工具调用最大尝试次数，默认 `1`。 |
| `--agent-retry-delay <seconds>` | 初始重试等待秒数，默认 `0`。 |
| `--agent-retry-backoff <factor>` | 重试退避倍率，默认 `2.0`。 |
| `--agent-retry-errors <keywords>` | 逗号分隔的可重试错误关键字。 |
| `--agent-preflight` | 输出模型配置预检结果，不调用网络 API，不需要输入文件。 |
| `--review-decision <json>` | 写入人工复核决定，并更新 run manifest。 |
| `--agent-validate-run <manifest-or-run-dir>` | 校验 run manifest、trace、output、review、decision 和 review index 是否完整。 |
| `--agent-export-report <path>` | 与 `--agent-validate-run` 配合，导出 Markdown 或 JSON 审计报告。 |
| `--agent-full-chain-test <output-dir>` | 运行无模型依赖的 fake 全链路验收，生成完整 run 产物和 audit 报告。 |
| `--agent-filing-plan <json>` | 从单个 agent 输出 JSON 或批量 run manifest 生成归档计划。 |
| `--agent-filing-root <dir>` | 归档目标根目录，配合 `--agent-filing-plan` 使用。 |
| `--agent-filing-action copy\|move` | 归档动作，默认 `copy`。 |
| `--agent-filing-apply` | 执行归档计划；未提供时只 dry-run。 |
| `--agent-filing-output <json>` | 把归档计划写入指定 JSON 文件。 |

trace JSONL 每行是一条步骤记录，主要字段包括：

```json
{
  "run_id": "agent-...",
  "file_path": "sample.pdf",
  "state": "TEXT_ANALYZED",
  "tool": "analyze_text",
  "input_summary": {},
  "output_summary": {},
  "error": "",
  "created_at": 1784040000.0
}
```

人工复核包包含 `file_path`、`agent_state`、`confidence`、`reasoning`、`trace`、`candidate_summaries`、`llm_analysis`、`vlm_analysis`、`error`、`created_at`，写入前会对 API key 等敏感片段做脱敏。

批量运行的 `manifest.json` 会记录每个文件的状态、置信度、trace、review、output 和错误摘要；`reviews/index.json` 会索引所有 `NEEDS_REVIEW` 与 `FAILED` 文件，便于集中复核。

模型配置预检：

```bash
python file_directory_predictor.py \
  --agent \
  --agent-preflight \
  --json
```

预检只检查 `AI_BASE_URL`、`AI_API_KEY`、`AI_MODEL` 是否完整，以及本机 PATH 中是否存在 legacy `z-ai`，不会请求远端模型。

恢复示例：

```bash
python file_directory_predictor.py \
  --agent \
  --resume ./logs/sample.trace.jsonl \
  --json
```

批量恢复示例：

```bash
python file_directory_predictor.py \
  --agent \
  --resume ./agent-runs/demo/manifest.json \
  --catalog ./catalog-mapping.xlsx \
  --json
```

批量恢复会把恢复后的文件 output、review package、manifest 和 review index 写回磁盘。已经是 `COMPLETED`、`NEEDS_REVIEW` 或 `FAILED` 的终态文件不会重跑。

人工复核决定示例：

```json
{
  "file_id": "review-123",
  "file_path": "input/review.txt",
  "decision": "approved",
  "final_prediction": {},
  "reviewer": "reviewer-a",
  "notes": "confirmed"
}
```

写入决定：

```bash
python file_directory_predictor.py \
  --agent \
  --resume ./agent-runs/demo/manifest.json \
  --review-decision ./decision.json \
  --json
```

运行产物验收：

```bash
python file_directory_predictor.py \
  --agent \
  --agent-validate-run ./agent-runs/demo/manifest.json \
  --agent-export-report ./agent-runs/demo/audit.md \
  --json
```

审计会检查：

- `manifest.json` 是否存在且可解析。
- manifest 中的状态统计是否与文件条目一致。
- 每个文件的 trace 和 output 是否存在。
- `NEEDS_REVIEW` / `FAILED` 文件是否有 review package。
- `reviews/index.json` 是否覆盖全部需要复核的文件。
- 已记录的人工复核 decision 文件是否存在。

无模型全链路验收：

```bash
python file_directory_predictor.py \
  --agent \
  --agent-full-chain-test ./test-reports/agent-full-chain-smoke \
  --json
```

该命令会生成一套 deterministic fake run：一个 `COMPLETED` 文件、一个 `NEEDS_REVIEW` 文件、一个 `FAILED` 文件、review index、review decision、audit Markdown 报告和 audit JSON 报告。它不调用真实模型，适合作为本地交付前的快速验收按钮。

归档计划 dry-run：

```bash
python file_directory_predictor.py \
  --agent \
  --agent-filing-plan ./agent-runs/demo/manifest.json \
  --agent-filing-root ./filed-cases \
  --agent-filing-output ./agent-runs/demo/filing-plan.json \
  --json
```

归档计划会把预测字段转换为目标路径：

```text
<agent-filing-root>/<predicted_case_type>/<predicted_volume>/<predicted_second_level_directory>/<predicted_material_category>/<source_file_name>
```

默认 dry-run 只生成计划，不复制或移动文件。计划 item 的主要字段包括：

```json
{
  "source": "input/起诉状.txt",
  "target": "filed-cases/民事一审案件/正卷/起诉状及相关材料/民事起诉状/起诉状.txt",
  "action": "copy",
  "status": "ready",
  "reason": "",
  "agent_state": "COMPLETED",
  "confidence": "high"
}
```

安全规则：

- 只有 `agent_state == COMPLETED` 且 `confidence != low` 的文件会进入 `ready`。
- `NEEDS_REVIEW`、`FAILED`、低置信、源文件不存在、目录字段不完整或目标文件已存在时，状态为 `blocked`。
- 路径片段会清洗路径分隔符和非法字符。
- 显式添加 `--agent-filing-apply` 后才会执行 `copy` 或 `move`。

执行 copy：

```bash
python file_directory_predictor.py \
  --agent \
  --agent-filing-plan ./agent-runs/demo/manifest.json \
  --agent-filing-root ./filed-cases \
  --agent-filing-action copy \
  --agent-filing-apply \
  --agent-filing-output ./agent-runs/demo/filing-plan.applied.json \
  --json
```

第三阶段行为和限制：

- 支持 batch agent mode，但仍只处理目录下的直接文件。
- 支持 partial resume；如果 trace 摘要不足以安全恢复，会返回明确失败原因。
- 工具调用接入 retry policy，CLI 可配置重试次数、退避间隔和可重试错误。
- run audit 只读取既有产物，不会修改 manifest 或重新执行工具。
- filing plan 只消费 agent 输出或 manifest，不会重新调用模型；默认不会改动文件系统。
- 真实模型 smoke test 需要配置 `AI_BASE_URL`、`AI_API_KEY`、`AI_MODEL`，或本机可用的旧 `z-ai` CLI fallback。

## 9. 内部处理流程

### 8.1 内容提取

`ContentExtractor.extract()` 根据扩展名提取：

```text
file_path
file_type
text
image_paths
page_count
extract_error
```

### 8.2 视觉分析

`VLMAnalyzer.analyze()` 对图片或 PDF 渲染页调用 `_run_zai_vision()`。

HTTP 模型路径使用：

```text
POST {AI_VISION_BASE_URL or AI_BASE_URL}/chat/completions
```

视觉请求使用 `image_url` data URI 形式，因此目标模型服务需要支持 OpenAI-compatible vision payload。

### 8.3 文本分析

`LLMAnalyzer.analyze_text()` 对提取文本调用 `_run_zai_chat()`，要求模型返回 JSON：

```json
{
  "doc_type_guess": "文书类型",
  "volume_guess": "正卷/副卷/未知",
  "case_clues": "案件类型线索",
  "key_phrases": "关键短语",
  "summary": "一句话摘要",
  "confidence": "high/medium/low"
}
```

### 8.4 候选召回

`DirectoryPredictor` 会把文件父目录名作为案号线索传给 `CatalogIndex.search_candidates()`。

`CatalogIndex.search_candidates()` 优先从案号识别案件类型（如 `民初`、`民终`、`刑初`、`行再`、`执`），并加载该案件类型下的所有编目规则。无法从案号识别案件类型时，才使用文书类型、案件线索、摘要、关键短语和文件名关键词进行候选召回。

### 8.5 候选精选

模型从候选编号中选择最匹配条目。解析时优先使用 `selected_index`，并用候选规则表反写字段，以减少模型自由生成不存在的目录。

## 10. 常用调参

在 `file_directory_predictor.py` 顶部可以调整：

```python
MAX_PDF_PAGES_FOR_VLM = 10
MAX_TEXT_CHARS = 6000
CLI_TIMEOUT = 180
```

| 参数 | 含义 |
| --- | --- |
| `MAX_PDF_PAGES_FOR_VLM` | PDF 前多少页送入视觉模型 |
| `MAX_TEXT_CHARS` | 文本模型最多接收的字符数 |
| `CLI_TIMEOUT` | 单次模型调用超时时间，单位秒 |

## 11. 常见问题

### 10.1 默认路径不适合当前部署环境

脚本顶部当前默认路径为：

```python
PROJECT_ROOT = Path("/Users/mooye/python project/smart-case-filing")
PROGRAM_DIR = Path(__file__).resolve().parent
DEFAULT_CATALOG = PROJECT_ROOT / "catalog-mapping.xlsx"
TMP_DIR = PROJECT_ROOT / "scripts" / "_tmp_predict"
```

如果部署到其他机器，建议运行时显式传入：

```bash
--catalog ./catalog-mapping.xlsx
```

如需长期使用，可把 `PROJECT_ROOT` 改为基于脚本目录或部署配置生成。

### 10.2 提示编目规则不存在

确认文件存在：

```bash
ls catalog-mapping.xlsx
```

并显式传入：

```bash
python file_directory_predictor.py sample.pdf --catalog ./catalog-mapping.xlsx
```

### 10.3 LLM 或 VLM 无返回

检查 HTTP 环境变量是否完整：

```bash
AI_BASE_URL
AI_API_KEY
AI_MODEL
```

或检查专用变量：

```bash
AI_CHAT_BASE_URL / AI_CHAT_API_KEY / AI_CHAT_MODEL
AI_VISION_BASE_URL / AI_VISION_API_KEY / AI_VISION_MODEL
```

开启调试输出：

```bash
export AI_DEBUG=1
```

PowerShell:

```powershell
$env:AI_DEBUG = "1"
```

### 10.4 PDF 识别不准

扫描 PDF 可能没有可提取文本，脚本会把前几页交给视觉模型。可以调大：

```python
MAX_PDF_PAGES_FOR_VLM = 5
```

### 10.5 长文档关键信息不在开头

脚本默认只取前 `MAX_TEXT_CHARS` 个字符。可以调大：

```python
MAX_TEXT_CHARS = 12000
```

## 13. 安全注意事项

- 不要提交真实 API key。
- 分享日志或报告前，确认 Authorization 已脱敏。
- 不要把敏感案件材料提交到公开仓库。
