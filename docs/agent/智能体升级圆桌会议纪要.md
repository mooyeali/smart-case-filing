# file_directory_predictor.py 智能体升级圆桌会议纪要

会议日期：2026-07-14  
会议目标：基于对 `file_directory_predictor.py` 的深度阅读，评估将“卷宗智能推理编目”从单次脚本预测升级为智能体的可行性，并形成后续开发决策。

## 1. 代码阅读摘要

当前 `file_directory_predictor.py` 是一个单文件批处理程序，约 1114 行，已经具备较清晰的流水线结构：

```text
输入文件
  -> ContentExtractor 提取文本、图片、表格内容
  -> VLMAnalyzer 生成视觉画像
  -> LLMAnalyzer 生成文本语义画像
  -> DirectoryPredictor 融合画像
  -> CatalogIndex 召回候选编目规则
  -> LLM 从候选编号中精选
  -> PredictionResult 输出结构化结果
```

主要模块和职责：

| 模块 | 当前职责 | 智能体化价值 |
| --- | --- | --- |
| `_get_model_config` / `_run_zai_chat` / `_run_zai_vision` | 模型配置、HTTP 调用和 z-ai fallback | 可抽为模型工具适配层，支持重试、审计、模型路由 |
| `CatalogIndex` / `CatalogLoader` | 加载和检索 `catalog-mapping.xlsx` | 可作为规则检索工具，供智能体多轮调用 |
| `ContentExtractor` | 提取 PDF、Word、图片、表格、文本内容 | 可作为材料读取工具，支持按需提取和失败恢复 |
| `VLMAnalyzer` | 分析图片/PDF 渲染页 | 可作为视觉分析工具，按需触发而不是固定串行执行 |
| `LLMAnalyzer` | 分析文本语义画像 | 可作为文本画像工具，支持补充提问和重新分析 |
| `DirectoryPredictor` | 固定编排、融合、候选匹配和结果组装 | 可拆为 agent orchestrator、state、policy、tools |
| `main` | CLI 单文件/批量入口、输出和日志保存 | 可保留为兼容入口，同时新增 agent run/review 子命令 |

现有测试覆盖范围较窄：

- `tests/test_catalog_index.py` 覆盖案号推导案件类型和父目录作为案号线索传入。
- `tests/test_cli_output.py` 覆盖显式/默认输出和日志文件写入。
- 尚未覆盖模型工具重试、状态机、人工复核、批量任务恢复、审计轨迹等智能体能力。

当前工作区风险：

- 当前分支是 `dev/agent`。
- 工作区存在未提交的非本次文档改动，包括 `catalog-mapping.xlsx` 删除、日志文件删除、`requirements.txt` 修改、`docs/.ignore/`、`logs/`、`offline_pkgs/linux/` 等。后续开发前应先确认这些改动是否保留，否则会影响运行验证。

## 2. 圆桌角色观点

### 产品负责人

当前脚本已能完成“单文件输入 -> 编目预测”的闭环，但用户真实工作流通常不是一次性预测，而是持续处理一批材料、发现不确定项、人工复核、重新分类、生成审计记录。智能体化的目标不应只是“把 LLM 多调几次”，而是把编目任务变成可解释、可暂停、可恢复、可人工确认的工作流。

产品侧建议：优先建设“编目任务智能体”，而不是泛化聊天智能体。第一阶段只解决材料编目工作流，不扩展到无边界问答。

### 架构负责人

当前代码已经有天然工具边界，但耦合在一个文件内。可行的升级路径是先做“受控智能体”：把已有能力注册为工具，建立 `AgentState` 和 `AgentRunner`，由策略决定是否调用文本分析、视觉分析、候选检索、候选精选、人工复核。

不建议一开始引入复杂多智能体框架。现有业务需要强约束、可审计、可复现，轻量自研状态机比直接套通用 Agent 框架更稳。

### 算法负责人

当前最大优点是候选约束：模型只能从候选编号中选，且最终字段回写候选规则。这一点必须保留。智能体化后可以增加三类能力：

1. 不确定性识别：当 `confidence=low`、模型解析失败、候选过多或文本/视觉冲突时进入复核。
2. 自我校验：检查 `selected_index`、案件类型、卷宗、材料类别与规则表是否一致。
3. 重试策略：对模型无返回、JSON 解析失败、候选为空等情况做有边界重试，而不是直接低置信结束。

算法侧不建议让模型自由生成编目路径。

### 数据与规则负责人

智能体应把 `catalog-mapping.xlsx` 当作权威规则源，不能把模型判断凌驾于规则表之上。后续可以增加规则版本号、规则加载摘要、候选召回轨迹和命中原因，便于复核为什么某个材料被分到某个目录。

当前 `CatalogIndex.search_candidates()` 已具备案号优先逻辑，这是智能体重要的规则工具。后续需要把“案号识别”“关键词召回”“候选精选”拆成可单独观察的步骤。

### 工程负责人

现有单文件脚本适合原型，但不适合继续承载智能体状态、工具注册、审计日志和恢复执行。建议做渐进式拆分：

```text
file_directory_predictor.py       # 兼容 CLI 外壳
smart_case_filing/
  config.py                       # 常量和模型配置
  models.py                       # dataclass / DTO
  model_client.py                 # OpenAI-compatible + z-ai fallback
  catalog.py                      # CatalogEntry / CatalogIndex / CatalogLoader
  extractors.py                   # ContentExtractor
  analyzers.py                    # LLMAnalyzer / VLMAnalyzer
  predictor.py                    # 原 DirectoryPredictor 兼容封装
  agent/
    state.py                      # AgentState / AgentStep
    tools.py                      # 工具接口和工具注册
    runner.py                     # 智能体状态机
    policy.py                     # 下一步决策策略
    review.py                     # 人工复核输入输出
```

但第一阶段不要一次性大重构。应先加测试，再抽出模型和状态，保持 CLI 结果兼容。

### 测试负责人

智能体化需要新增以下测试层级：

- 单元测试：状态转换、工具输入输出、候选校验、重试策略。
- 合同测试：旧 CLI 输出字段保持兼容。
- 无模型测试：用 fake model client 固定返回，验证流程。
- 回归测试：保留 `selected_index` 回写候选字段的安全约束。
- 文档校验：README/使用说明链接不能断。

测试侧建议第一阶段不要依赖真实 API，所有智能体流程先用 fake tool/model 完成确定性验证。

### 安全与合规负责人

材料可能包含真实案件信息，智能体化后会产生更多中间状态和日志，因此必须默认脱敏或最小记录。建议：

- `AgentTrace` 记录步骤、工具名、输入摘要、输出摘要，不默认记录全文材料。
- API key 不进入日志、状态文件或测试报告。
- 人工复核文件应可以独立保存，避免覆盖原材料。
- 所有模型响应都必须标记模型名、时间、prompt 类型、解析结果。

### 运维负责人

智能体必须支持断点恢复，否则批量处理时成本不可控。建议引入每个文件一个 `run_id`，每步状态写入 JSONL 或 SQLite。第一阶段可以用 JSONL 简化实现，后续再升级数据库。

运维侧还建议将当前硬编码 `PROJECT_ROOT`、`TMP_DIR` 改为配置对象，避免部署到其他机器时需要改代码。

## 3. 可行性结论

结论：可行，且适合采用“受控任务型智能体”路线。

可行依据：

1. 现有流程已经有清晰的工具边界：提取、视觉分析、文本分析、候选检索、候选精选、输出。
2. 现有输出已经结构化为 `PredictionResult`，便于扩展为 `AgentState`。
3. 现有候选约束机制能抑制模型编造，是智能体化后必须保留的安全基础。
4. 当前 CLI 可作为兼容入口，新增 agent 模式不会破坏既有用法。
5. 测试虽少，但已有 fake/patch 模式，适合继续扩展为无模型的确定性测试。

关键前提：

- 不做开放式聊天智能体；先做“编目任务智能体”。
- 不允许模型自由写最终目录；最终字段仍必须来自规则表。
- 第一阶段不引入外部 Agent 框架，使用本地状态机和工具接口实现。
- 先完成模块边界与状态可观测，再扩展多轮决策。

## 4. 建议目标形态

智能体的最小目标不是替代当前预测器，而是增强当前预测器：

```text
用户提交一个文件或目录
  -> Agent 创建任务状态
  -> 按需调用内容提取、视觉分析、文本分析、候选检索、候选精选工具
  -> 对低置信/冲突/解析失败做有限重试或转人工复核
  -> 输出 PredictionResult + AgentTrace + ReviewDecision
  -> 支持批量任务恢复和审计
```

第一阶段必须保留：

- `python file_directory_predictor.py <file> --catalog ... --json`
- `python file_directory_predictor.py --batch <dir> --catalog ... --json`
- 当前 JSON 顶层字段。

第一阶段新增：

- `--agent`：启用智能体状态机。
- `--trace <path>`：保存步骤轨迹。
- `--review-output <path>`：低置信或需人工复核时输出复核材料。
- `--resume <trace>`：从上次轨迹恢复未完成任务。

## 5. 主要风险和缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 单文件继续膨胀 | 后续维护困难 | 渐进拆包，每次保留 CLI 兼容测试 |
| 模型决策失控 | 编目字段被编造 | 继续强制 `selected_index`，最终字段从候选回写 |
| 批量处理不可恢复 | 大批材料失败成本高 | 引入 `AgentTrace` JSONL 和 `run_id` |
| 日志泄露材料 | 合规风险 | 默认记录摘要，不记录全文和密钥 |
| 测试依赖真实模型 | 不稳定且昂贵 | fake model client + fake tool runner |
| 当前工作区有删除项 | 开发验证被污染 | 开发前恢复或明确处理 `catalog-mapping.xlsx` 等未提交改动 |

## 6. 会议决议

1. 通过智能体升级可行性评估。
2. 采用“受控任务型智能体 + 工具接口 + 本地状态机”路线。
3. 第一阶段不做开放式聊天、不引入复杂多 Agent 框架。
4. 保留当前 CLI 和 JSON 输出兼容性。
5. 优先补测试和抽象模型客户端，再引入 `AgentState`、`AgentTool`、`AgentRunner`。
6. 输出升级开发计划，作为后续实施依据。
