# 智能体升级第四阶段圆桌会议纪要

会议日期：2026-07-15

会议目标：评估第三阶段完成后的批量可恢复智能体是否已经是完整且符合需求的智能体；如果不是，确定第四阶段开发范围并进入开发。

## 1. 当前状态复盘

第三阶段已经完成“批量可恢复的编目任务 agent”：

| 能力 | 当前实现 | 验证证据 |
| --- | --- | --- |
| 批量 agent | `--agent --batch` 处理目录直接文件 | `tests/test_agent_batch_cli.py` |
| run manifest | `smart_case_filing/agent/run_manager.py` 写入 `manifest.json` | `tests/test_agent_run_manager.py` |
| 独立 trace/output/review | 每个文件独立路径 | `tests/test_agent_batch_cli.py` |
| partial resume | 单文件和 manifest resume | `tests/test_agent_partial_resume.py` |
| retry policy | `smart_case_filing/agent/retry.py` | `tests/test_agent_retry.py` |
| review index | `reviews/index.json` | `tests/test_agent_run_manager.py`、`tests/test_agent_review.py` |
| fake full-chain | completed / needs-review / failed / resume | `tests/test_agent_full_chain.py` |

当前测试基线：

```text
python -m unittest discover -s tests
Ran 48 tests
OK
```

## 2. 结论

会议结论：当前系统已经是“可测试的批量编目任务 agent”，但还不是完整且符合需求的交付级智能体。需要开启第四阶段开发。

原因：

1. 批量命令接受 `--review-output`，但当前实现仍固定写入 run directory 下的 `reviews/`，参数语义不完整。
2. batch resume 当前只输出恢复结果，不会把恢复后的文件状态回写 manifest，也不会重建 review index。
3. retry policy 已有代码，但没有 CLI 参数，真实运行时无法按任务配置重试次数、退避间隔和可重试错误。
4. 真实模型 smoke 前缺少配置预检，用户只能在运行失败后从错误信息推断是 HTTP 模型配置缺失还是 legacy `z-ai` 不可用。
5. 复核材料仍停留在“生成 review package”，没有 review decision 的落地格式；人工修正结果无法进入 run manifest 形成闭环。

## 3. 圆桌角色观点

### 产品负责人

第三阶段已经能跑一批文件，但用户还需要“可运营”的闭环：批量结果在哪里、失败怎么复核、复核后如何留下最终决定。第四阶段应补齐 CLI 参数语义、恢复持久化和复核决定记录。

### 架构负责人

第四阶段不应大改状态机。建议只做运行管理增强：

- `AgentRunManager` 支持外部 review directory。
- batch resume 复用 `AgentRunManager.record_file()` 回写 manifest。
- 新增 review decision helper，避免把人工复核决定塞进 trace。

### 工程负责人

当前 retry policy 默认一次尝试，等同于没有运行时可配置能力。应新增 CLI 参数：

```text
--agent-retry-attempts
--agent-retry-delay
--agent-retry-backoff
--agent-retry-errors
```

默认值保持兼容，用户不传时行为不变。

### 测试负责人

第四阶段测试重点：

- batch `--review-output` 确实改变 review package 和 index 位置。
- batch resume 会更新 manifest 和 review index。
- CLI retry 参数会传入 runner 并触发重试。
- 模型配置预检在未配置时返回结构化结果，不依赖真实网络。
- review decision 文件会被脱敏写入并可索引到 manifest。

### 安全与合规负责人

人工复核决定可能包含案件信息和模型错误摘要，写盘前必须继续脱敏 API key。manifest 中只记录 decision 文件路径和状态，不写入长正文。

## 4. 第四阶段建议范围

必做：

1. 完整实现 batch `--review-output` 目录语义。
2. batch resume 回写 manifest、output 和 review index。
3. 将 retry policy 暴露为 CLI 参数。
4. 增加模型配置 preflight，输出 HTTP 模型配置和 legacy fallback 可用性。
5. 增加 review decision 文件格式和 manifest 索引。
6. 更新文档和 fake full-chain 报告。

暂不做：

1. Web UI。
2. 数据库持久化。
3. 递归目录处理。
4. 在线人工审批系统。
5. 多 Agent 框架。

## 5. 会议决议

1. 当前功能不是最终完整智能体，但已经具备第四阶段增强所需的基础。
2. 第四阶段目标定义为“可运营的批量编目 agent 闭环”。
3. 继续保持轻量 CLI 和 JSON 文件工作流，不引入服务端依赖。
4. 第四阶段完成后再开启最终全链路测试。
