# 智能体升级第七阶段圆桌会议纪要

会议日期：2026-07-16

会议目标：评估第六阶段后的智能体是否已经完整符合“智能卷宗归档”需求；如果完整，则开启全链路测试；如果仍有缺口，则确定第七阶段开发范围。

## 1. 当前状态复盘

第六阶段已经补齐可重复、无模型依赖的全链路验收入口：

| 能力 | 当前实现 | 验证证据 |
| --- | --- | --- |
| 单文件 agent 执行 | `--agent <file>` | `tests/test_agent_cli_integration.py` |
| 批量 agent 执行 | `--agent --batch <dir>` | `tests/test_agent_full_chain.py` |
| 执行轨迹 | trace JSONL | `smart_case_filing/agent/runner.py` |
| run manifest | `AgentRunManager` | `tests/test_agent_run_manager.py` |
| 低置信复核包 | `ReviewPackageWriter` | `tests/test_agent_review.py` |
| partial resume | `--resume <manifest>` | `tests/test_agent_resume.py` |
| 人工复核决定 | `--review-decision` | `tests/test_agent_full_chain.py` |
| run 审计与报告 | `--agent-validate-run` / `--agent-export-report` | `tests/test_agent_audit.py` |
| 无模型全链路验收 | `--agent-full-chain-test` | `tests/test_agent_full_chain_cli.py` |

当前系统已经能够完成“识别、预测、复核、审计、验收”的闭环，但它仍主要停留在结果产物层面。

## 2. 结论

会议结论：当前 agent 还不是完整的智能卷宗归档智能体。它能够判断材料应该归入哪里，也能够生成可审计 run 产物，但还缺少把预测结果转换为可执行归档动作的能力。

缺口：

1. 预测结果没有被转换为标准化的目标目录路径。
2. 批量 run manifest 不能直接生成归档计划。
3. 用户缺少安全的 dry-run 归档演练入口。
4. 复核未通过、低置信、失败材料需要在归档计划中被明确阻断，而不是混入自动归档。

因此本轮不开启最终全链路测试完成结论，而是进入第七阶段。

## 3. 圆桌角色观点

### 产品负责人

用户最终关心的不是 JSON 里预测了什么，而是这些文件能否安全进入对应卷宗目录。第七阶段至少要给出“将要把哪个文件放到哪里”的计划。

### 工程负责人

建议新增独立 helper，不改现有预测、批量、复核、审计流程。输入可以是单个预测 JSON 或 run manifest，输出归档计划。默认只生成计划和 dry-run，只有显式 apply 才复制或移动文件。

### 测试负责人

测试需要覆盖：

1. 单个预测结果生成目标路径。
2. run manifest 生成多文件归档计划。
3. `NEEDS_REVIEW`、`FAILED`、低置信文件不自动归档。
4. dry-run 不改动文件系统。
5. apply copy 能落地文件。

### 运维负责人

归档目标目录必须由用户显式传入，不能写死。计划文件应记录 action、source、target、status、reason，便于审计和人工复核。

## 4. 第七阶段范围

必做：

1. 新增归档计划 helper。
2. 新增 CLI，支持从 agent 输出或 run manifest 生成归档计划。
3. 默认 dry-run，不移动或复制文件。
4. 支持显式 apply copy / move。
5. 更新测试、使用说明和阶段报告。

暂不做：

1. Web UI 拖拽归档。
2. 数据库任务队列。
3. 与法院正式业务系统对接。
4. 自动覆盖同名文件。
5. 对低置信或失败材料自动归档。

## 5. 会议决议

1. 第七阶段目标定义为“可审计归档计划与安全演练执行”。
2. 第七阶段不改变模型调用、预测流程和上层业务流程。
3. 所有自动归档动作默认 dry-run。
4. 只有 `COMPLETED` 且置信度非 low 的材料可以进入可执行 copy / move 计划。
5. 第七阶段完成后，系统才能从“可验收预测智能体”推进到“可协助归档智能体”。
