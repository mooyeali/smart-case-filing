# 智能体升级第二阶段圆桌会议纪要

会议日期：2026-07-14  
会议目标：基于第一阶段已完成的智能体基础设施，讨论第二阶段需要完成的工作，形成可执行的开发范围和验收标准。

第一阶段已经完成以下能力：

| 能力 | 当前实现 | 验证情况 |

当前测试基线：

```text
python -m unittest discover -s tests
Ran 14 tests
OK
```

## 2. 第二阶段目标

第二阶段目标是把“智能体骨架”接入真实编目工具链，让 `--agent` 模式具备可用的单文件端到端能力，同时保持旧 CLI 行为兼容。

目标状态：

```text
python file_directory_predictor.py ./sample.pdf \
  --catalog ./catalog-mapping.xlsx \
  --agent \
  --trace ./logs/sample.trace.jsonl \
  --review-output ./logs/sample.review.json \
  --json
```

应完成：

1. 使用真实工具完成内容提取、文本分析、视觉分析、候选召回、候选精选。
2. 写入 JSONL trace，每一步只记录摘要，不默认记录全文材料。
3. 低置信、模型解析失败、候选为空或工具失败时输出 review package。
4. `--agent` 输出仍兼容普通预测 JSON 的主要字段，同时附带 agent 状态和 trace 路径。
5. 支持 `--resume <trace>` 的最小恢复能力，至少能读取已有 trace 并跳过已完成步骤或给出明确不可恢复原因。

## 3. 圆桌角色观点

### 产品负责人

第二阶段必须从“可演示”走向“可用”。用户不会关心内部状态机是否优雅，而是关心：

- `--agent` 是否真的能处理一个文件。
- 低置信时是否能生成可交给人工复核的材料。
- trace 是否能解释为什么做出这个分类。

产品侧建议第二阶段只承诺单文件 agent 端到端，不扩展复杂批量恢复。批量 agent 可以作为第三阶段。

### 架构负责人

第一阶段已经建立 `AgentRunner` 和工具注册表，但目前 runner 使用固定工具名顺序。第二阶段可以继续保守推进：

```text
extract_content
analyze_visual
analyze_text
retrieve_candidates
select_catalog
finalize_prediction
write_review_if_needed
```

不建议立即做复杂 policy 引擎。可以先把顺序 runner 扩展为“条件步骤”：

- 没有图片时跳过 `analyze_visual`。
- 文本为空时跳过 `analyze_text`，但保留 trace。
- `confidence=low` 时进入 `NEEDS_REVIEW`。

### 算法负责人

第二阶段必须继续保留候选约束机制。最终字段仍应来自候选规则，不允许模型自由生成目录。

需要新增两个校验：

1. `selected_index` 必须在候选范围内。
2. 最终输出字段必须与候选条目完全一致。

如果模型返回的 `case_type` 与候选不一致，应记录到 trace，但不能覆盖规则表字段。

### 数据与规则负责人

候选召回应成为可观察工具，trace 至少需要记录：

- 输入关键词摘要。
- 案号或父目录线索。
- 候选数量。
- 前若干个候选的材料类别和案件类型。
- 是否使用了 fallback 候选。

这能帮助定位“模型判断对了，但候选召回错了”的问题。

### 工程负责人

第二阶段不应大规模拆 `file_directory_predictor.py`。建议新增 adapter 层：

```text
smart_case_filing/agent/legacy_tools.py
```

该文件负责把旧类包装成 agent tools。这样可以减少主脚本修改范围，并为后续拆包留接口。

建议新增：

- `build_legacy_tool_registry(catalog, model_client=None)`
- `run_agent_prediction(args)`
- `prediction_result_to_agent_output(...)`

### 测试负责人

第二阶段测试必须保持不依赖真实模型。建议新增：

- `tests/test_agent_legacy_tools.py`
- `tests/test_agent_cli_integration.py`
- `tests/test_agent_resume.py`

测试策略：

- 用 fake model client 固定返回文本分析和候选精选结果。
- 用临时 catalog entries 构造小型规则表。
- 验证 trace JSONL 行数、状态、工具名和摘要。
- 验证 low confidence 会写 review JSON。
- 验证普通 CLI 测试仍通过。

### 安全与合规负责人

第二阶段开始接入真实工具链后，trace 泄露风险上升。必须规定：

- `input_summary` 不记录完整文本，只记录长度、文件类型、页数、图片数。
- `output_summary` 记录模型解析后的结构化摘要，不记录完整 prompt。
- review package 可以包含更多上下文，但必须脱敏 API key。
- 所有异常信息进入 trace 前先过 `redact_secret`。

### 运维负责人

`--trace` 路径必须稳定可控。没有传入时可默认写入：

```text
PROGRAM_DIR / "agent_trace.jsonl"
```

但建议第二阶段改为按 run_id 写独立文件：

```text
logs/agent/<run_id>.trace.jsonl
```

如果本阶段时间有限，至少需要确保：

- trace 父目录自动创建。
- 已存在 trace 时追加或 resume 行为明确。
- 写 trace 失败时不静默吞掉。

## 4. 第二阶段范围

### 必做

1. 新增 `smart_case_filing/agent/legacy_tools.py`。
2. 将旧能力注册为 agent tools：
   - `extract_content`
   - `analyze_visual`
   - `analyze_text`
   - `retrieve_candidates`
   - `select_catalog`
   - `finalize_prediction`
3. 扩展 `AgentRunner` 支持视觉步骤和最终输出组装。
4. `_run_agent_cli(args)` 从占位实现改为真实执行。
5. `--trace` 写入真实 JSONL。
6. `--review-output` 在低置信或失败时写入 review package。
7. 新增无模型集成测试，验证 agent mode 单文件流程。
8. 保持旧 CLI 和 JSON 输出兼容。

### 可选

1. `--resume` 支持从 trace 中恢复到下一步。
2. 支持 batch agent mode。
3. 把 `CatalogEntry`、`FileContent`、`PredictionResult` 拆入 `smart_case_filing/models.py`。
4. 增加 retry policy。

### 暂不做

1. 开放式聊天智能体。
2. 多 Agent 框架。
3. SQLite 持久化。
4. Web UI。
5. 模型自动路由。

## 5. 建议任务拆分

### Task A: Legacy Tool Adapter

新增 `smart_case_filing/agent/legacy_tools.py`，包装旧类能力。

验收：

- fake 文件输入可调用 `extract_content`。
- fake model client 可驱动 `analyze_text` 和 `select_catalog`。
- `retrieve_candidates` 返回候选摘要。

### Task B: Agent Runner V2

扩展 runner 支持：

- visual step 可跳过。
- finalize prediction。
- low confidence -> `NEEDS_REVIEW`。
- failed tool -> `FAILED`。

验收：

- trace 中出现完整工具链步骤。
- 正常路径返回 `COMPLETED`。
- 低置信路径返回 `NEEDS_REVIEW`。

### Task C: Real CLI Agent Mode

把 `_run_agent_cli(args)` 接入真实 runner。

验收：

- `--agent --json` 输出 JSON。
- `--trace` 写 JSONL。
- `--review-output` 在低置信时写 JSON。
- 普通非 agent CLI 不受影响。

### Task D: Minimal Resume

实现 `--resume <trace>` 的最小语义：

- 能读取 trace。
- 如果最后状态是 `COMPLETED`，直接返回已完成摘要。
- 如果最后状态是 `FAILED` 或 `NEEDS_REVIEW`，返回明确状态，不重复执行。
- 若需要真正从中间恢复，留到第三阶段。

验收：

- resume completed trace 不重复调用工具。
- resume failed trace 输出失败摘要。

### Task E: Documentation and Test Report

更新中英文使用说明，补充真实 agent mode 行为和限制。

验收：

- `python -m unittest discover -s tests` 通过。
- Markdown 本地链接检查通过。
- 新增一份 agent mode fake run 测试报告。

## 6. 验收标准

第二阶段完成时必须满足：

1. `--agent` 不再是占位实现。
2. 单文件 agent mode 能完成端到端预测。
3. trace JSONL 至少包含：
   - `STARTED`
   - `EXTRACTED`
   - `TEXT_ANALYZED` 或跳过说明
   - `VISUAL_ANALYZED` 或跳过说明
   - `CANDIDATES_RETRIEVED`
   - `MATCHED`
   - `COMPLETED` / `NEEDS_REVIEW` / `FAILED`
4. 最终目录字段仍来自候选规则表。
5. 低置信或失败能生成 review package。
6. 所有测试通过。
7. 普通 CLI 输出字段兼容。

## 7. 主要风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| Agent 输出与旧 JSON 不兼容 | 破坏已有调用方 | 新增 agent 字段时不删除旧字段 |
| trace 记录过多原文 | 合规风险 | 默认摘要化，review package 单独控制 |
| 工具 adapter 过度耦合旧脚本 | 后续拆包困难 | adapter 只做薄包装，业务逻辑仍逐步迁移 |
| resume 语义过度承诺 | 实现复杂度上升 | 第二阶段只做最小 resume |
| fake 测试与真实模型差距 | 线上不稳定 | 保留一份手工真实模型 smoke test 文档 |

## 8. 会议决议

1. 第二阶段目标确定为“真实单文件 agent mode 端到端可用”。
2. 不做开放式智能体，不做 batch agent，不做复杂 resume。
3. 优先通过 `legacy_tools.py` 接入旧能力，避免一次性大拆包。
4. trace 和 review package 是第二阶段核心交付，不是附属功能。
5. 继续坚持测试先行和 fake model client，真实模型只做 smoke test。
6. 第二阶段完成后再讨论第三阶段：批量 agent、断点恢复、重试策略和模块拆包。
