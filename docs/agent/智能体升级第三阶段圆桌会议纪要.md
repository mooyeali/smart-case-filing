# 智能体升级第三阶段圆桌会议纪要

会议日期：2026-07-15

会议目标：基于当前 `dev/agent` 分支的第二阶段实现，评估现有功能是否已经是一个完整且符合需求的智能体；如果是，则进入全链路测试；如果不是，则形成第三阶段开发范围。

## 1. 当前实现复盘

当前实现已经完成第二阶段目标：`--agent` 不再是占位入口，单文件模式可以通过 `AgentRunner` 调用旧编目工具链，生成 trace JSONL，并在低置信或失败时写入 review package。

已具备能力：

| 能力 | 当前实现 | 验证证据 |
| --- | --- | --- |
| 单文件 agent 入口 | `file_directory_predictor.py --agent` 调用真实 `AgentRunner` | `tests/test_agent_cli_integration.py` |
| 旧能力工具适配 | `smart_case_filing/agent/legacy_tools.py` 注册提取、分析、候选召回、候选精选、结果组装工具 | `tests/test_agent_legacy_tools.py` |
| 状态机执行链路 | `smart_case_filing/agent/runner.py` 记录 `STARTED` 到终态的步骤 | `tests/test_agent_runner.py`、`tests/test_agent_runner_v2.py` |
| trace JSONL | `AgentTraceStore` 追加步骤记录，summary 会过滤内部字段 | `tests/test_agent_state.py`、runner 测试 |
| 低置信复核 | `confidence=low` 进入 `NEEDS_REVIEW` 并写 review package | `tests/test_agent_cli_integration.py` |
| 最小 resume | 读取已有 trace 的最后状态并返回摘要，不重复执行工具 | `tests/test_agent_resume.py` |
| 复核包结构化 | review package 包含文件、状态、置信度、候选摘要、模型分析和错误信息 | `tests/test_agent_review.py` |
| 非 agent 兼容 | 普通 CLI 输出和日志行为继续通过测试 | `tests/test_cli_output.py` |

当前测试基线：

```text
python -m unittest discover -s tests
Ran 34 tests
OK
```

## 2. 结论

会议结论：当前功能是“第二阶段可用的单文件受控编目 agent”，但还不是完整且符合最终需求的智能体。

理由：

1. 当前 agent 只覆盖单文件任务，不支持 batch agent mode，无法满足真实卷宗材料批量处理场景。
2. 当前 `--resume` 是最小摘要语义，不能从中间步骤继续执行，断点恢复能力还不完整。
3. 当前没有 retry/backoff policy，模型调用、解析失败、候选为空等场景只能直接失败，无法稳定支撑长任务。
4. 当前 trace 默认可以追加到同一个文件，但没有 run 级目录、run manifest、trace/review/output 的统一归档约定。
5. 当前全链路验证仍主要依赖 fake model 和单元测试，真实模型 smoke 会受本机外部模型命令或配置影响，还没有形成可重复的全链路测试门禁。
6. 当前主脚本仍承担大量业务类和 CLI 编排，后续 batch、resume、重试继续叠加会扩大维护风险。

因此，本轮不直接宣布进入“完整智能体全链路验收完成”，而是开启第三阶段开发计划。

## 3. 圆桌角色观点

### 产品负责人

第二阶段已经能演示单文件闭环，但用户真实工作流是成批处理卷宗材料。一个完整智能体至少需要批量运行、失败复核列表、可恢复任务和清晰的运行报告。第三阶段应把“可用单文件 agent”推进到“可运行一批材料的编目任务 agent”。

验收重点：

- 能用 `--agent --batch` 或等效入口处理目录下多个文件。
- 每个文件有独立状态、trace 和 review 结果。
- 运行结束能输出批量摘要，区分 `COMPLETED`、`NEEDS_REVIEW`、`FAILED`。

### 架构负责人

当前顺序 runner 适合第二阶段，但第三阶段不能继续把所有恢复逻辑堆在 CLI 函数里。建议新增轻量 run 层：

```text
AgentRunManager
  -> 为每个文件创建 run_id
  -> 决定 trace/review/output 路径
  -> 调用 AgentRunner
  -> 汇总 run manifest
```

第三阶段仍不引入复杂多 Agent 框架，保持受控状态机和旧能力 adapter，但需要把运行管理、resume 策略、复核包写入从 CLI 中拆出来。

### 算法负责人

第二阶段已经保持“最终目录字段来自候选规则”的安全边界。第三阶段需要增强失败可解释性：

- 候选为空时记录关键词、案号、fallback 情况。
- 模型返回非法候选编号时进入可重试或复核路径。
- 低置信、解析失败和候选冲突要有明确 reason code。

### 数据与规则负责人

批量模式下必须保留每个文件的候选召回摘要，否则复核人员无法判断是模型判断问题还是规则表召回问题。建议 run manifest 至少记录：

- 输入文件路径。
- 最终状态。
- 预测目录字段。
- 置信度。
- trace 路径。
- review 路径。
- 错误摘要。

### 测试负责人

第三阶段需要从“单元测试覆盖 agent 组件”升级为“全链路测试覆盖任务级行为”。建议新增：

- fake model batch agent 测试。
- partial resume 测试。
- retry policy 测试。
- manifest 结构测试。
- 真实模型 smoke 测试脚本或文档化门禁，允许在未配置模型时明确跳过。

### 安全与合规负责人

批量 trace 会放大数据泄露风险。第三阶段必须继续默认只记录摘要，并为 review package 明确边界：

- trace 不记录完整原文。
- review package 继续脱敏 API key。
- manifest 不写入全文。
- 失败堆栈或模型错误写入前必须脱敏。

### 运维负责人

第三阶段需要稳定的运行目录约定。例如：

```text
agent-runs/<run_id>/
  manifest.json
  traces/<file_id>.trace.jsonl
  reviews/<file_id>.review.json
  outputs/<file_id>.json
```

这样批量任务、恢复任务和问题排查都有稳定入口。

## 4. 第三阶段建议范围

必做：

1. 批量 agent mode。
2. run manifest 与运行目录管理。
3. partial resume，从最后成功步骤之后继续执行。
4. retry/backoff policy，覆盖模型调用、解析失败和临时工具错误。
5. 批量复核包索引和运行摘要。
6. 可重复的 fake full-chain 测试。
7. 文档化真实模型 smoke test。

暂不做：

1. 开放式聊天智能体。
2. 多 Agent 协作框架。
3. Web UI。
4. SQLite 或服务端数据库持久化。
5. 模型自动路由和成本优化。

## 5. 会议决议

1. 当前功能不是完整最终形态，但已经是可用的第二阶段单文件 agent。
2. 不直接开启“最终全链路验收完成”；先启动第三阶段开发计划。
3. 第三阶段目标定义为“批量可恢复的编目任务 agent”。
4. 第三阶段仍沿用轻量受控状态机，不引入通用 Agent 框架。
5. 第三阶段完成后再进行覆盖批量、resume、retry、review 和真实模型 smoke 的全链路测试。
